# Acme Corp — Enterprise Mock Scenario

## Company Profile

**Acme Financial Technologies** is a mid-size fintech company providing
digital banking services. They are building AI agents to improve customer
support efficiency while maintaining strict regulatory compliance.

## Team

- **Platform Team** (manages Plato + agent infrastructure)
- **Customer Experience Team** (building the support agent)
- **Compliance Officer** (defines data handling rules)

## The Agent: Acme Support Bot

A customer-facing support agent that:
- Answers account balance and transaction history queries
- Processes refund requests (with approval workflow)
- Escalates complex issues to human agents
- Remembers customer preferences across sessions

## Risk Classification: **Tier 1** (High Risk)

Handles: PII (names, account numbers, transaction data), financial operations

## Applicable Policies

All 4 policies apply:
- `security-standards.md` — credential management, input validation, injection defense
- `agent-architecture.md` — Strands patterns, memory isolation, tool design
- `compliance-requirements.md` — PII masking, audit logging, human approval for refunds
- `testing-standards.md` — 80% unit test coverage, integration tests, evaluation suite

## Specific Compliance Rules

1. **Refund threshold**: Refunds > $500 require human approval
2. **PII in memory**: Customer names stored as first initial + last name only
3. **Transaction data**: Never stored in agent memory; queried live from API
4. **Audit trail**: Every refund request logged with: customer_id, amount, approved_by, timestamp
5. **Investment advice**: Agent must refuse any investment-related questions with standard disclaimer

## Expected CLAUDE.md (generated by Plato)

When a developer on the CX team asks Plato to help build this agent,
Plato should generate a CLAUDE.md that includes:

1. Platform coding standards (Strands SDK, hooks, tool patterns)
2. All Tier 1 policies (security + architecture + compliance + testing)
3. Acme-specific rules (refund threshold, PII masking, investment disclaimer)
4. Readiness checklist derived from policies
5. Policy version tags for drift detection

## Demo User Journey

```
Developer: "I'm building a customer support agent for Acme's banking app"

Plato: [Phase 1]
  → Asks: "What data will it handle?"
  → Developer: "Account balances, transaction history, refund processing"
  → Plato classifies as Tier 1 (financial PII)
  → Creates GitHub repo with:
    - CLAUDE.md (all policies + Acme-specific rules)
    - .github/pull_request_template.md
    - .github/ISSUE_TEMPLATE/
    - docs/SPEC.md
    - Readiness checklist

Developer: [Phase 2]
  → Uses CC to build the agent
  → CC reads CLAUDE.md → follows security + compliance rules automatically
  → Asks Plato: "How should I handle refund approvals?"
  → Plato: "Use ToolPolicyHook + human-in-the-loop pattern. Here's the code..."

Plato: [Phase 2.5]
  → Breaks spec into GitHub issues
  → Issue #1: "Implement account balance query tool"
  → Issue #2: "Implement refund tool with $500 approval threshold"
  → Issue #3: "Add PII masking to memory storage"

Developer: [Phase 3]
  → CC implements Issue #1 → pushes PR
  → Plato reviews:
    ✅ Tool uses @tool decorator with proper types
    ❌ Missing input validation on account_id
    ❌ Raw customer name stored in memory (PII violation)
  → Creates Issue #4: "Fix PII masking in account balance tool"
  → Developer's CC reads Issue #4 → fixes → pushes
  → Plato re-reviews → ✅ Approved → Merges

Plato: [Phase 4]
  → Runs readiness checklist
  → All checks pass → generates deploy config
  → Developer deploys → smoke test passes

Plato: [Phase 5]
  → Monitors agent health
  → Detects: 3 refund requests > $500 auto-approved (policy violation!)
  → Alerts CX team → patches ToolPolicyHook
```
