## Project Overview

Building a customer-facing support agent for Acme Financial Technologies'
digital banking platform. Handles account queries, refund processing, and
issue escalation.

**Risk Tier: 1 (High)** — Handles PII and financial data.

## Architecture Requirements

### Framework
- Use **Strands SDK** as the agent framework
- Register all tools with `@tool` decorator, proper docstrings and type hints
- Use hook middleware for cross-cutting concerns:
  - `SoulSystemHook` — agent personality
  - `MemoryHook` — conversation context
  - `AuditHook` — **REQUIRED** for all Tier 1 agents
  - `GuardrailsHook` — input/output validation
  - `ToolPolicyHook` — tool access control + refund approval workflow
  - `CompactionHook` — long session summarization

### Memory
- Use AgentCore Memory (not local files or databases)
- Configure namespace isolation: `/actors/{actorId}/`
- **NEVER** store raw PII in memory. Mask customer names to first initial + last name
- Transaction data must be queried live from API, never cached in memory
- Configure all 4 LTM strategies: Semantic, UserPreference, Summary, Episodic

### Session Management
- Session IDs: UUID format (minimum 33 characters)
- Thread-based session IDs for multi-turn conversations
- Implement agent pooling with LRU eviction

## Security Requirements

### Credentials
- NO hardcoded API keys, tokens, or passwords anywhere in code
- Use AWS Secrets Manager for all third-party credentials
- Use IAM roles with temporary credentials for AWS services

### Input Validation
- Validate all user inputs (account IDs, amounts, dates)
- Implement input length limits (max 2000 characters per message)
- Reject inputs with known injection patterns
- Never pass raw user input into system prompts

### Error Handling
- Never expose internal errors, stack traces, or system info to users
- Log errors with full context internally
- Return user-friendly error messages
- Implement circuit breakers for banking API calls

## Compliance Requirements (Tier 1)

### PII Handling
- Customer names: store as "J. Smith" not "John Smith"
- Account numbers: never store; use tokenized references
- Transaction data: query live, never persist in agent memory
- Implement data minimization: only collect what's needed

### Human Approval Workflow
- **Refunds > $500**: MUST require human approval before processing
- Use ToolPolicyHook to enforce the threshold
- Log all approval decisions: customer_id, amount, approved_by, timestamp
- Implement escalation path when approver is unavailable

### Audit Logging
- Log every tool invocation (timestamp, actor_id, tool, input summary, output)
- Use AuditHook (automatic)
- Retain logs for minimum 12 months
- Include session_id in all log entries

### Prohibited Responses
- **NEVER** provide investment advice of any kind
- Standard disclaimer: "I'm not qualified to provide investment advice. Please consult a licensed financial advisor."
- Do not speculate about market conditions, stock prices, or financial trends
